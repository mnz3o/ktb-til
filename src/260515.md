## 과제 피드백
- 상속을 결정하는 핵심 단어는 is-a 관계
- 상속은 행위의 확장이 아니라 개념의 확장이어야 한다
- 분류 혹은 조합으로 관계를 나눌 수 있다
- 클래스 관계가 분류라면 상속 사용
- 조합 관계라면 필드(속성)로 표현
- class explosion를 고려하자
- 클래스의 속성을 많이 사용하자

## 객체란 무엇일까?
- 눈에 보이는 모든 것을 객체라고 볼 수 있다

## 객체 지향으로 우리가 얻는 것
- 재사용성 -> 잘 만든 부품을 다른 곳에서도 사용할 수 있다

- 유지보수성 -> 오류 수정과 기능 수정이 편해진다

- 대규모 협업 -> 객체 단위로 역할을 나누어 여러 명이 협업하기 좋다

## 다형성
- 하나의 타입으로 여러 객체를 다룰 수 있도록 하는 것
- 동일한 코드가 상황에 따라 다른 동작을 수행할 수 있도록 해준다

### 오버로딩
- 하나의 클래스 안에 같은 이름의 메서드를 매개변수의 개수나 타입을 다르게 정의
- 메서드, 생성자에 적용 가능함
- 같은 동작을 수행하는 메서드를 일관된 이름으로 정의 + 다양한 상황 유연 대처
- 사용이유 -> 코드의 일관성과 가독성을 높일 수 있기 때문이다.

- 비슷한 일들을 하는 메서드들의 이름을 하나로 통일해서 얻는 편의성

### 오버라이딩
- 부모 클래스의 메서드를 자식 클래스에서 재정의하여 동작을 바꾸는 개념
- 클래스 상속 시 부모 클래스의 필드와 메서드를 물려받음
- 물려받은 메서드가 자식 클래스의 상황과 안맞거나 동작을 바꾸고 싶을 때 사용
- 자식 클래스에서 메서드 이름, 매개변수, 리턴 타입을 유지하면서 본문만 바꿔 정의
- 부모 클래스 = 상위의 개념 = 더 포괄적
- 부모의 기본 동작을 기반으로 자식 클래스 상황에 맞게 재정의하는 객체 지향 방식

## 캡슐화
- 객체의 내부 상태를 외부에서 접근하지 못하도록 숨김
- 메서드(검증 된 통로)를 통해 간접적으로 접근하도록 하는 객체지향 개념
- 캡슐화가 없으면, 우리 프로그램은 무법지대가 된다
- 데이터 보호, 유효 검사도 안된다.
- 내부 구현을 은닉하지 못한다.
- getter, setter
- 특정 필드에 대해서 값을 읽는 메서드(getter), 값을 설정하는 메서드(setter)

## 인터페이스
- 구현 세부사항 없이 클래스가 구현해야 할 메서드의 집합만을 정의한 설계 명세서

#### 사용 시 얻게 되는것
- 유연성
- 병렬 개발

### 인터페이스와 상속의 차이점
#### 상속
- is-a 관계
- 수직적, 계층적 구조
- 부모의 기능을 물려받아 재사용
- 단일 상속만 가능(부모는 하나)
- 비유: "생물학적 혈통"

#### 인터페이스
- can-do 관계
- 수평적, 기능적 확장
- 클래스에 특정 자격/능력 부여
- 다중 구현 가능(자격증은 여러개)
- 비유: "전문 자격증"
- 확장성과 유연성을 갖는다

##### 공통점은 상속 묶고, 차이점(능력) 인터페이스로 분리한다

## 제네릭
- 자바는 타입에 대해 매우 엄격하다
- 포괄적인
- 특정 타입 하나에 얽매이지 않고
- 여러 타입을 두루 담을 수 있는
- "범용 주머니"

#### 기술적 설명
- 데이터 타입을 고정하지 않고
- 클래스나 메서드를 "사용할 때" 이 때, 데이터 타입을 결정하게 하는 것

## Why
- 코드를 작성할 때는 이게 어떤 타입이 들어올 지 모를 때
- 모든 객체의 조상인 오브젝트를 받아서 사용할 수 있다
- 꺼낼 떄 문제발생 -> 타입 변환

### 제네릭이란걸 만들자
- 잘못된 타입이 들어오면 실행 전에 에러 발생시키고
- 타입 변환도 생략하자

- Object -> 일단 대충 담고 나중에 생각하자
- 제네릭 -> 처음부터 따져서 넣고, 여기에 뭐가 들어올지 어느정도 준비하고, 사고 안내려고 하겠다
- 타입의 안정성을 높이겠다

### 제한된 타입
- 너무 타입이 자유로운 제네릭은 위험하다.

### 해결방법
- T에 들어올 수 있는 클래스는 반드시 부모클래스 이거나, 부모클래스의 자식이여야 한다.
- 엉뚱한 타입이 들어오는걸 컴파일 단계에서 차단할 수 있다
- 기능도 활용할 수 있다
- 그냥 제네릭은 여기에 뭐가 들어올 지 모른다.
- 그래서 뭐가 들어올지 모르니, 객체에 메서드나 필드를 접근할 수 없다

#### 자바는 자유보단 안전을 추구한다. 완전히 자유로운 제네릭은 실제로는 많이 사용하지 않는다

## 컬렉션
- 여러 개의 데이터를 쉽고 효과적으로 처리할 수 있도록 자바에서 미리 만들어둔 "표준화된 자료구조 세트"

### list
- 순서가 있는 배열
#### 특징
- 데이터를 넣은 순서가 유지된다.
- 동일한 데이터가 들어와도 중복을 허용한다.
- 비유: 음악 재생 목록 
- 실무 예시 - 게시판의 게시물 목록, 채팅 메시지 이력

### set
- 중복 없는 집합
#### 특징
- 순서를 보장하지 않는다.
- 중복을 허용하지 않는다.
- 비유: 학생들의 학번
- 실무 - 방문자 IP, 특정 글의 좋아요를 누른 유저의 ID

### map
- 키와 값이 한 쌍인 자료 구조
#### 특징
- 키(열쇠)와 value(값)를 한 쌍으로 묶어서 저장한다
- 키는 중복을 허용하지 않는다
- value는 중복을 허용한다
- 키를 기반으로 검색하는게 매우 빠르다
- 실무 - 유저의 로그인 정보 
 -> 로그인 키 값은 1, 밸류 "홍길동", 로그인 키 값 2, 밸류 "홍길동" 허용

### 어떤걸 사용할까
- 데이터를 순서대로 쌓고 꺼낼 때 > list > 인덱스를 통한 빠른 접근이 가능함
- 중복을 체크하거나 제거할 때 > set > contains() 연산이 매우 빠르고 중복을 원천 차단함
- 특정 이름으로 데이터를 찾아야 할 때 > map > 방대한 데이터 중 원하는 데이터를 빛의 속도로 찾음

## Thread (스레드)
- 하나의 프로그램 내에서 동시에 실행될 수 있는 독립적인 실행 흐름 단위

### 사용이유
- 여러 작업을 동시에 실행해 애플리케이션의 성능과 응답성을 높이기 위해서 사용함

### 멀티스레드와 싱글스레드
- 비유 - 요리사 음식점
- 싱글 스레드 - 스레드가 한 개인 것 > 요리사 한명이 주문, 손질, 서빙
- 멀티 스레드 - 스레드가 두개 이상인 것 > 한 요리사는 주문만, 다른 요리사는 손질만 
- 독립적으로 움직이는 손이 여러개인 상태
- "하나의 프로그램 안에서 여러개의 실행 줄기가 동시에 뻗어나가는 것"

## 왜 사용하냐
- 빠르게 하는게 주목적 아니다.
- "멈추지 않고, 동시에 수행하기 위해서"
- 기본적으로 메인스레드 (메서드를 실행하면 만들어지는 흐름 = 메인 스레드)
- 응답성 유지 -> 고화질 영상 다운받으면서, 동시에 채팅 보내기
- 백그라운드 처리
- 자원 효율 극대화
- 대용량 데이터 배치 작업

### 주의사항
- 스레드가 고독사 -> 스레드 내부에서 에러가 나면 메인 프로그램 알려지지 않고, 스레드만 사라짐
- 가시성 문제
- 순서가 꼬일 수 있다 -> 순서 중요한 경우는 이 순서가 안 지켜질 수 있다.

## Runnable
- 새로운 실행 흐름 작업의 정의서다
- 스레드를 만들어서 새로운 실행흐름으로 프로그램 실행시키기

### 실행 제어와 로직의 분리 
1. 스레드 상속
- 스레드 생성, 시작, 종료
- 실제 코드 흐름
- 비지니스 로직
- 한 클래스 섞여있다.

2. 러너블로 구현하면
- "무슨 일을 할지"만 정의하면 된다.
- 러너블은 "인터페이스"

### 스레드와 러너블
- thread (일꾼) - 실제 실행 흐름을 만들고 제어
- runnable (업무 지시서) - 무엇을 해야한다고 적혀 있는 종이

### 사용방법
- 람다식을 많이 씀

## 실무에서는 멀티 스레드를 어떻게 사용하녀

## ExecutorService
### new thread의 위험성
1. 스레드는 일반적으로 직접 생성하지 않는다
- 생성 비용이 높다
- 스레드 <- 실행흐름
- os가 관리하는 무거운 자원이라서
- 만들고 버릴 때마다 cpu+mem 소모가 높다
2. 리소스 고갈 문제(oom)
- 사용자 1명이 다운로드 요청할 때마다 파일을 내려주는 서버
3. 관리하기가 힘들다

#### 우리는 스레드를 매 번 만들지 않는다.
- 스레드 <- 일꾼, 일을 하는 실행하는 흐름

### 핵심 구조
- 작업 제출 -> 개발자가 "무슨 일을 할지 정의 한 것 -> runnable" 이거만 던진다
- 블로킹 큐 -> 일꾼이 부족하면 작업들이 순서대로 대기하는 대기실
- 스레드 풀 -> 실제 작업을 하는 일꾼들

### 왜 사용하는거?
1. 자원보호
- 스레드를 직접 생성하면 메모리가 부족해질 수 있다
2. 성능 향상
- 스레드 생성, 소멸 <- OS 입장에 무겁고 비싼 작업
3. 관리하기 편하다
- 스레드가 직접 생성하면 먼저 요청한게 뭔지 뭔지도
- 끝나면 어디로 반환될지 
- 관리하기가 힘들다

### 스레드 풀
- 작업을 처리하기 위한 스레드를 매번 새로 생성하지 않고, 미리 만들어 둔 스레드 집합을 재사용하여 작업을 처리하는 구조

### 현업에서 스레드풀
- @Async

## Synchronized
- 여러 스레드가 동시에 공유 자원에 접근하지 못하도록 일시적으로 해당 자원에 대한 접근을 하나로 제한하는 동기화 키워드

### 임계 영역
- 공유 자원(힙)에 접근하는 프로그램 코드의 일부분을 의미함

## Lock
- 동시에 여러 스레드가 하나의 공유 자원에 접근하지 못하도록 막는 장치

- 싱크로나이즈드는 <- 대기하고 있는 스레드가 누가 먼저 온지 모른다.
- 기다리는 조건이 단일 하나다.
- 무한히 대기할 수도 있다.
- 융통성이 없는 자동문


- Lock은 스레드들이 누가 먼저 왔는지 알 수 있다. <- 먼저 온 스레드를 먼저 처리할 수 있다.
- 대기하는 조건을 걸 수 있다.
- 무한 대기를 안 할 수도 있다.

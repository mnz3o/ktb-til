## Git / GitHub

### Git
- 컴퓨터 파일의 변경 사항을 추적하고,
- 여러 사용자가 파일 작업을 함께 조율하기 위한 분산 버전 관리 시스템

### GitHub
- Git 저장소 호스팅을 지원하는 대표적인 오픈소스 웹 서비스

## 버전 관리 시스템
- 프로그램을 수정하고 개선하면서 버전을 관리하는 시스템
- 변경될 때마다 버전 번호를 증가시킴
- 어떤 프로그램이 몇 번 수정되었는지 관리

## 중앙 집중식 버전 관리 시스템
- 중앙 서버에서 버전을 관리하는 방식
- 모든 사용자가 중앙 서버를 기준으로 작업

### 장점
- 다른 사람들과 동시에 동일한 자료를 볼 수 있다
- 로컬 컴퓨터의 저장 공간을 아낄 수 있다
- 작업 순서를 알 수 있어 이전 기록으로 돌아가기 쉽다
- 파일 유실 가능성이 상대적으로 적다

### 단점
- server에 인터넷 연결이 안되면 작업할 수 없다
- 서버가 손상되면 복구하기 어렵다
- 동시에 같은 내용을 작업하기 어렵다

## 분산 버전 관리 시스템
- 프로젝트의 전체 히스토리와 버전 관리 정보를 각 사용자의 로컬 시스템에 복제하여,
- 네트워크에 의존하지 않고도 버전 관리를 수행할 수 있도록 하는 시스템

### 장점
- 작업순서를 알 수 있어 이전 기록으로 돌아가기 쉽다
- server에 인터넷 연결이 잠시 안되어도 작업을 할 수 있다
- 서버가 손상되어도 복구 가능성이 높다
- 동시에 같은 내용을 작업할 수 있다

### 단점
- 관리가 어렵다

## 명령어
- git은 working directory, stage, history 영역으로 구성된다
- working directory 안에는 tracked files, untracked files가 있다

- git init > Git 저장소 생성
- git add > untracked files에서 stage로 이동
- git commit > stage 내용을 history로 저장
- git push > 변경사항을 서버에 업로드

- git remote > 로컬 저장소와 연결된 원격 저장소를 관리하는 명령어
- git status > 현재 작업 중인 저장소 상태 확인

- git add . / git add * 는 지양

- git add .  
  -> 현재 디렉터리 기준으로 변경된 모든 파일을 재귀적으로 stage에 추가

- git add *
  -> 숨김파일(.gitignore, .env 같은 것)은 기본적으로 제외될 수 있음

- git clone은 원격 저장소에 있는 레포지토리를 처음 복제해올 때 사용하는 명령어

- git pull은 원격 저장소에 있는 레포지토리의 최신 내용을 로컬 저장소로 가져올 때 사용하는 명령어

- git pull = fetch + merge

: pull을 하고 push를 하라

- git log은 기록을 볼 수 있는 명령어
- git diff는 변경 사항을 확인할 수 있음

- status는 무엇이 바뀌었는지 어떤 파일이 변경되었는지,
- diff는 어떻게 바뀌었는지 어떤 내용이 수정되었는지 확인

- git difftool은 변경사항을 GUI 도구로 확인할 때 사용하는 명령어
- git show은 commit 정보 볼 때 사용하는 명령어
- git blame은 수정 내역을 확인할 때 사용하는 명령어
- git stash은 임시저장, 최근 커밋과의 차이점을 보관

- git branch은 코드의 다른 버전을 관리하기 위해 사용되는 독립적인 작업 흐름이다

- git checkout은 메인 branch에서 다른 branch로 넘어갈 때 사용

- git bisect은 히스토리를 이진 탐색으로 좁혀 최초 불량 커밋을 찾는다

- git restore은 commit하지 않은 변경사항을 마지막 상태로 되돌린다

- git revert는 되돌아가는데 되돌아간다는 기록을 남긴다

- git reset은 작업하던 곳을 이전 상태로 되돌려주는 기능이지만, 웬만하면 사용하지말자

- .gitignore는 Git에서 추적하지 않을 파일이나 폴더를 지정하는 파일

## Merge 방식
- 3-way merge
  -> 공통 조상 commit을 기준으로 병합하는 방식

- fast-forward merge
  -> branch 이동만으로 병합하는 방식

- squash and merge
  -> 여러 commit을 하나로 합쳐 병합하는 방식

## Branch
- Main/Master
  -> 안정성이 보장되며 운영 중 발생할 수 있는 치명적인 문제에 대비해 롤백할 수 있는 커밋 집합 유지

- Develop(Staging)
  -> 로컬에서의 테스트와 개발을 마친 후 최종적인 테스트가 이루어지는 브랜치

- Feature
  -> Develop 브랜치를 기점으로 새 브랜치를 만들어 개발

## 객체 지향 프로그래밍
- 실체화(객체화)되어서 재활용하자

## 클래스
- 속성과 기능, 생성자가 하나로 묶여져 있는 설계도
- 관련된 속성과 메서드를 묶어서 객체를 생성하기 위한 템플릿

## 상속
- 부모 클래스의 속성과 기능을 자식 클래스가 물려받아 재사용하는 것

## 함수
- 재활용하기 위해 만들어둔 코드들의 집합

## 한줄정리
- 클래스 : 관련된 속성과 메서드를 묶어서 객체를 만들기 위한 설계도

- 메서드 : 객체가 수행할 수 있는 동작을 정의하며, 하나의 작업을 캡슐화하여 호출만으로 반복 사용할 수 있도록 만든 코드 블록

- 생성자 : 클래스의 객체가 생성될 때 자동으로 호출되어 객체의 초기화를 수행하는 특수한 메서드

- 객체 : 클래스를 기반으로 메모리에 생성된 실제 데이터(실체)

- 상속 : 부모 클래스의 속성과 기능을 자식 클래스가 물려받아 재사용하는 것
